# ai2nao 设计系统

## 产品定位

ai2nao 是一个本机桌面数据工作台，不是营销网站，也不是移动端应用。

设计决策优先服务：

- PC 桌面端、本机浏览器使用。
- 高密度但可读的数据检查。
- 在不同本机数据源之间稳定切换。
- 对新鲜、过期、错误、本地写入等状态给出清晰信任信号。

移动端访问明确不在范围内。窄桌面窗口需要避免布局破碎，但不做手机优先的堆叠设计。

## 页面类型

### 简单清单页

适用于主任务是扫描一组扁平列表的页面：

- 下载
- Mac 应用
- Homebrew 包
- 本地模型库存

默认结构：

```text
页面头
筛选工具条
全宽表格或列表
分页 / 次级诊断信息
```

### 复杂工作台页

适用于用户先选择对象，再查看详情或执行操作的页面：

- Chrome 域名分析
- GitHub 雷达
- 对话 / session 浏览器
- 仓库详情工作流
- 目录或项目 drilldown

默认结构：

```text
页面头
轻上下文工具条
左侧索引 / 队列
右侧工作区
主任务 tab
次级详情区
```

## 复杂工作台布局

复杂页面的标准结构：

```text
+--------------------------------------------------------------------+
| 页面头：标题、用途、数据状态、主操作                                |
+--------------------------------------------------------------------+
| 轻工具条：当前对象、搜索、范围、更多筛选                              |
+--------------------+-----------------------------------------------+
| 左侧索引            | 右侧工作区                                     |
| 360-400px           | 当前对象详情和主任务                            |
+--------------------+-----------------------------------------------+
| 次级详情：全局统计、调试信息、旧视图                                  |
+--------------------------------------------------------------------+
```

规则：

- 左侧是方向盘，只负责选择域名、队列、repo、session、项目等对象。
- 右侧是工作台，只解释和操作当前选中的对象。
- 复杂页默认应该选中一个有价值对象，而不是展示空态或泛泛总览。
- “全部对象”视图需要是左侧索引里的一级选项，例如“全部域名”或“全部线索”。
- 页面级状态必须出现在页面头或工具条里，不能埋在长列表底部。

## 宽度与间距

全局内容框架：

```text
max-width: 1760px
horizontal padding: 32px
```

除非页面类型有强理由变化，否则不要让单个路由变宽或变窄。用户切换页面时，应用不应该跳。

复杂工作台默认栅格：

```text
grid-template-columns: 360-400px minmax(0, 1fr)
gap: 24px
```

密集排行索引用 380px，例如 Chrome 域名。轻量队列可用 360px。

## 工具条

使用轻上下文工具条，不使用厚重筛选卡片。

第一行可以放：

- 当前对象。
- 主搜索。
- 日期或范围摘要。
- 更多筛选入口。
- 清空或重置，仅在存在 active filters 时显示。

低频控制进入“更多筛选”展开的内联第二行：

- 高级日期字段。
- 类型 / kind 筛选。
- 粒度 / 聚合控制。
- profile、root、config 字段。
- rebuild 或诊断操作。

不要把重要控制放在长列表下方。

## 指标与摘要

指标首先解释当前对象。

复杂页面规则：

- 当前对象指标放在右侧工作区顶部。
- 左侧索引行只保留最少指标，通常是支持排序的一个数字。
- 全局统计进入次级详情或折叠区。
- 首屏不要放会抢主任务的全局指标卡。

例子：

- Chrome 域名：左侧行显示域名 + 访问数；右侧摘要显示访问、首次、最近。
- GitHub 雷达：左侧行显示线索 + 优先级信号；右侧摘要显示 health、score、证据数、关联 repo 数。

## 卡片

卡片只用于独立对象：

- 一个 repo。
- 一条线索。
- 一条访问记录。
- 一个 session。

不要把卡片当成通用页面区块。由一堆卡片堆出来的页面会像生成式 dashboard，而不是工作台。

页面结构优先使用边框、横向区块、表格、折叠区和稳定栅格。

## 主任务 Tab 与次级详情

Tab 只放真正的工作模式，不要变成所有次级内容的垃圾桶。

适合放 tab 的内容：

- 访问明细 vs 时间矩阵。
- 当前线索 vs 复盘队列。
- 列表 vs 证据视图，如果两者都是主任务。

适合放折叠区的内容：

- 全局统计。
- freshness 内部状态。
- rebuild 详情。
- legacy view。
- 调试指标。

## 状态设计

每个复杂页面都需要明确这些状态：

- loading。
- empty。
- error。
- fresh。
- stale。
- partial。
- 旧数据仍可见时的 refreshing。

用户必须能判断：

- 数据是不是当前最新。
- 旧数据还能不能参考。
- 哪个操作能修复当前状态。
- 操作只写本地 ai2nao 数据，还是会触碰外部服务。

## URL 状态

URL 状态需要区分系统默认和用户显式选择。

Chrome 域名页示例：

- 无 domain 参数：系统默认使用 Top 1 域名。
- `domains=github.com`：用户显式选择了某个域名。
- `scope=all`：用户显式选择全部域名。

不要让“空参数”同时表示“系统默认”和“明确查看全部”。

## 文案

使用工具型语言：

- 这个区域是什么。
- 用户能做什么。
- 数据处于什么状态。
- 下一步动作会做什么。

避免情绪化、营销式和过长说明。这个产品靠清楚、稳定和可信来建立信任。

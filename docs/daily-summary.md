# 每日工作摘要设计说明

本文档沉淀 `ai2nao` 的「每日工作摘要」功能设计结论。它不是实现说明书，而是产品与架构约束文档，供后续 `/gstack-plan-eng-review` 和实际开发参考。

若未来实现与本文档冲突，以明确的新决策为准，并及时回写本文档。

---

## 1. 功能目标

把已有的两类本地证据拼起来：

- **当天做过什么**：来自 Atuin 当日命令历史
- **这些行为大概属于什么项目**：来自已索引 repo、README、manifest

目标不是生成一段好看的 AI 文案，而是做出一个**可信的单日工作回放**：

- 用户能快速知道今天主要在干什么
- 用户能看到明天大概率从哪里接着做
- 用户能感知这一天更像是在推进、排障、探索，还是被频繁打断

这个功能首先服务于**用户自己每天回顾**，而不是日报导出或团队绩效跟踪。

---

## 2. 产品定位

这是一个 **本机优先、事实优先、推断透明** 的工作回放层。

它不是：

- 监控工具
- 绩效工具
- 全知全能的工作日志机器人

它更像：

- 一个基于 shell 历史和 repo 语义的回顾助手
- 一个未来可扩展成“工作记忆层”的单日切片

---

## 3. 首版范围

### 3.1 首入口

首个可见入口放在 **`/atuin` 页面**，而不是先做独立 CLI。

原因：

- `/atuin` 已经是“按日回看”的天然容器
- 用户心智更顺，像是在命令历史之上加一层解释，而不是另起一个孤立功能

### 3.2 触发方式

首版采用 **显式触发**：

- 用户点击“生成摘要”或“刷新摘要”时才运行
- 不在选中日期时自动偷偷生成
- 需要保留一个**明确开关**，便于快速关闭该功能

### 3.3 输出形态

首版输出不是纯 markdown，而是 **结构化 payload**。

自然语言只是其中一部分，推荐至少包含：

- `summary`
- `nextUp`
- `workMode`
- `fragmentation`
- `degraded`
- `degradeReason`
- `facts`
- `meta`

UI 上表现为：

- 一张位于命令表上方的**辅助卡片**
- 不是页面主英雄区
- 原始命令列表仍然是事实主视图

---

## 4. 已接受的扩展范围

以下内容已明确纳入当前设计方向：

### 4.1 明日接力棒

摘要中包含 `Next up / 明日接力棒` 区块。

目的：

- 把回顾变成第二天进入上下文的桥
- 提高这个功能“每天愿意打开一次”的概率

### 4.2 今日模式判断

摘要中包含 `Today felt like` 一类的轻量工作模式判断，例如：

- implementation
- debugging
- exploration

注意：这是**推断层**，不是事实层。

### 4.3 上下文切换 / 碎片化提示

摘要中包含一个轻量的“今天是否高度碎片化”的信号。

它的价值不是复述命令，而是帮助用户意识到：

- 今天是否沉浸在一个项目里
- 还是被来回切换打断了一整天

---

## 5. 延后到 Phase 2 的内容

以下内容已明确 **defer**，但它们不是“有空再说”，而是较明确的下一阶段方向：

### 5.1 证据可回看层

摘要下方增加可回看的证据，例如：

- 命中的 repo
- 代表命令
- 时间跨度
- 置信提示

这是未来把“会说”升级成“可信”的关键层。

### 5.2 跨天工作线程

把单天摘要串成跨天脉络，识别：

- 昨天开始、今天继续、明天还会继续的工作

这是未来把“单日摘要”升级成“工作记忆层”的关键一步。

---

## 6. 数据与推断边界

### 6.1 Required input

最小输入是：

- 单个本地日历日内的 Atuin entries

日界定义：

- 与现有 Atuin 日视图一致
- 使用 `ai2nao serve` 所在环境的本地时区语义

### 6.2 Required derived signal

必须先得到：

- 基于 `cwd` 的 repo 归属

当前设计选择：

- v1 不满足于纯最长前缀匹配
- 使用更强的路径解析 / repo root 归属逻辑
- 但该逻辑只在**首次生成某天摘要时**执行
- 归属结果与摘要一起缓存

若高精度归属失败：

- 允许 fallback
- 但 fallback 必须**显式可观测**
- 禁止 silent fallback

### 6.3 Optional context

可选语义上下文来自：

- README
- manifest

它们只能以**预压缩的短 blurb** 形式进入摘要流程，不能拿大段原文直接塞进 prompt。

若 README / manifest：

- 不可读
- 编码异常
- 解析失败
- 太大
- 超时

则一律视为**无额外上下文**，而不是半用半不用。

### 6.4 事实与推断的权力边界

必须坚持：

- **事实骨架先由确定性逻辑计算**
- **LLM 只负责文本层**

也就是说：

- repo 命中
- facts
- degraded
- degradeReason
- generation meta
- fragmentation 信号

优先由确定性逻辑产出。

LLM 主要负责：

- `summary`
- `nextUp`

若自然语言与事实冲突：

- **facts 永远赢**
- 文本应被丢弃或降级

---

## 7. 降级策略

这是本功能最重要的信任边界之一。

### 7.1 弱信号日

若当天信号过弱，不应硬写故事。

系统必须退化为：

- factual recap
- 极简事实卡片

而不是“看起来像总结”的幻觉文本。

### 7.2 空白日

即使活动很少，也不应让功能区域忽然消失。

推荐行为：

- 显示一张极简 recap 卡片
- 明确写出“未检测到显著活动”或同等语义

### 7.3 模型失败

若 LLM：

- 超时
- 不可用
- 返回空
- 返回畸形结构

系统必须：

- 保留 factual recap + facts
- 明确提示自然语言摘要不可用

不能整块空白。

### 7.4 事实冲突

若模型文本与 facts 冲突：

- 视为一次失败
- 丢弃文本
- 回到 factual recap

### 7.5 stale 缓存

若已有旧缓存，而本次刷新失败：

- 默认显示这次生成出的 factual recap
- 明确标记摘要刷新失败
- 旧成功摘要只作为次级可回看信息

---

## 8. 缓存策略

首版采用**轻量按日缓存**，但不能只按日期缓存。

缓存必须绑定关键输入版本，例如：

- `date`
- `model`
- `promptVersion`
- repo / 索引相关版本指纹
- 归属规则版本

原因：

- 否则会出现最糟糕的 silent staleness
- 用户看到的是旧理解，却以为是最新结果

同时：

- 页面可以先显示 stale 结果
- 但必须明确标识“正在更新”
- 不能把旧缓存伪装成最新结果

---

## 9. Prompt 与隐私策略

### 9.1 Prompt 预算

进入模型前必须有**硬预算控制**，例如：

- Top-N repo
- 每 repo 代表命令上限
- repo blurb 长度上限
- 总字符 / token 上限

没有预算时，这个功能会在“最忙的一天”表现最差。

### 9.2 脱敏时机

所有命令在进入模型前必须先脱敏。

并且：

- 缓存不能偷偷保留原始危险值
- 日志不能偷偷保留原始危险值

### 9.3 脱敏策略

默认采取：

- **白名单 / 意图压缩优先**
- 少量高危模式黑名单补刀

而不是“先把全量命令送进模型，再看输出时遮一遮”。

### 9.4 Prompt 持久化

默认不持久化 prompt 原料。

落盘应只保存：

- 结果
- 版本元数据
- 必要的指纹

若未来要调试 prompt，应通过显式 debug 开关，而不是默认长期保留。

---

## 10. 观测性要求

这个功能不能是“坏了只能靠感觉猜”的系统。

### 10.1 降级原因码

所有降级都应有明确的结构化 reason code，例如：

- `sparse_signal`
- `repo_match_ambiguous`
- `llm_timeout`
- `llm_malformed`
- `text_fact_conflict`

这些 reason code 应尽量进入：

- payload
- logs
- metrics

### 10.2 生成元数据

结果中应带最小版本元数据，例如：

- `generatedAt`
- `model`
- `promptVersion`
- `resolverVersion`

哪怕默认不全部展示给用户，也必须可用于调试和观测。

---

## 11. 测试要求

首版不接受“主要靠人工点点看”。

### 11.1 必须有 ship-block 级主线测试

至少应有一条集成主线，覆盖：

- Atuin 输入
- repo 命中
- 结构化 payload
- 正常生成
- 弱信号降级
- 模型失败降级

### 11.2 LLM contract test

除常规 mock 外，还需要覆盖至少这些边界：

- 正常结构
- 空响应
- 畸形结构
- 超时 / 失败

### 11.3 日期切换竞态测试

既然首入口在 `/atuin`，必须单独验证：

- 快速切换日期时
- 过期响应不会覆盖当前选中日

---

## 12. UX 约束

### 12.1 信息层级

`/atuin` 页面中：

- 原始命令仍是事实层
- 摘要卡片是高价值增强层
- 摘要不应抢走页面主导权

### 12.2 推断语气

UI 和文案必须明确表现“这是推断层”。

例如应允许：

- `看起来像`
- `可能在做`
- `根据今天的 shell 活动推测`

不应把它表现成：

- 审计日志
- 确凿事实
- 无误真相

---

## 13. 长期方向

这次设计已明确：

- 每日摘要不是 disposable text
- 它是未来工作记忆层的资产

因此后续路线可明确理解为：

### Phase 1

- 单日摘要
- 结构化 payload
- 明日接力棒
- 今日模式判断
- 碎片化提示

### Phase 2

- 证据可回看层
- 跨天工作线程

### 更远方向

- 基于摘要资产做周报、检索、长期回放

---

## 14. 当前仍需注意的风险

- `cwd` 到 repo 的高精度归属可能在 monorepo、symlink、多根目录场景下复杂化
- `Today felt like` 容易过度解释，必须接受“很多天应该省略”
- `fragmentation` 与 `context-switch` 的对外表述要保持单一、稳定
- “无意义活动 / 稀疏活动”的阈值仍需在实现规范里落成具体样例

---

## 15. 一句话总结

这个功能的正确方向不是“让 AI 会写日报”，而是：

**用本地事实骨架托住一层诚实的推断，让用户每天愿意看一眼，并愿意明天继续用它接着干活。**

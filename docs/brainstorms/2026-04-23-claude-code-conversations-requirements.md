# Claude Code 本地对话读取（第一版）

## Problem Frame

本仓库已支持只读读取 Cursor 本地聊天记录（SQLite + 解析层 + Web 与 CLI）。使用 Claude Code 的用户在 `~/.claude` 下同样有本地会话轨迹（按项目组织、以 JSONL 等形式落盘），但目前无法在 ai2nao 中统一浏览与校验这些内容。第一版目标是在**不依赖云端**的前提下，补齐「发现项目 → 发现 JSONL → 可靠解析 → 在网页上可刷新重扫」的闭环，并与现有 Cursor 能力在**产品心智**上对齐（本地 AI 对话总览）。

## Requirements

**发现与配置**

- R1. 系统应能从用户本机默认路径发现 Claude Code 项目目录列表，默认根路径为当前用户下的 `~/.claude/projects`（或与之等价的已展开绝对路径）。
- R2. 应支持可选的「自定义根路径」参数（与 Cursor 的 `dataPath` 类能力同理），以便在不同安装位置、多用户或调试时指向另一目录；未指定时使用默认路径。
- R3. 「项目」在此指 `projects` 根目录下的直接子目录（每个子目录视为一个逻辑项目）；第一版不要求递归扫描 `~/.claude` 下 `projects` 以外的其它文件夹。

**JSONL 与解析**

- R4. 在每个项目目录下，应识别并枚举用于会话/轨迹的 `.jsonl` 文件（第一版以扩展名为准；同名或临时文件策略在实现阶段按实机目录约定处理）。
- R5. 解析器应对 JSONL **逐行**读取：每行一条独立 JSON；必须正确处理 UTF-8、行尾差异、以及文件末尾可选换行。
- R6. **解析正确性（第一版核心）**：对 Claude Code 写入的已知行类型，应解析为明确的结构化结果（字段可映射到统一「事件/消息」模型或保留 `type` + `payload` 的判别联合类型），**不得静默丢弃**可识别字段；对无法识别的行，应保留原始 JSON 文本（或原始对象）并标记为未知，以便排查与后续扩展，而不是静默跳过。
- R7. 单行 JSON 损坏（非完整 JSON、截断）时，该行应产生可诊断的错误信息（行号、文件路径），且不影响其余行的解析。
- R8. 解析结果应能通过自动化测试验证：仓库内应包含**基于真实结构脱敏或合成的 JSONL fixture**，覆盖多行、多事件类型、损坏行、空行（若实机可能出现则须定义策略）等情形。

**API 与行为**

- R9. 应提供与现有 `cursor-history` 风格一致的只读 HTTP API（具体路径在规划阶段命名），至少包括：列出项目、列出某项目下的 JSONL 文件、获取某文件的解析结果（或分页/流式策略在规划阶段确定）。
- R10. 所有读取均为只读；不得修改 `~/.claude` 下任何文件。

**Web 界面**

- R11. 网页端应提供**刷新**控件：用户触发后重新执行「项目列表 + 文件列表」的发现逻辑（不要求整页重载；可与 React Query `refetch` 等效）。
- R12. 网页端应能展示解析后的内容，使用户能确认「解析是否完整、结构是否合理」（第一版以清晰可读为先，视觉不必与 Cursor 页逐像素一致）。
- R13. **信息架构应对齐现有 Cursor 历史页**：项目（工作区）列表 → **会话**列表 → **消息**时间线；底层虽为 JSONL 文件，但应对用户隐藏「按文件挑会话」的细节，除非实现上必须提供文件级排障入口（可作为次要链接或开发模式）。
- R14. 由于 Claude Code 侧会话边界可能由 JSONL 事件序列推导，第一版须定义稳定的「会话」标识与排序规则（例如按时间、按 thread id 字段等），并在 UI 上与 Cursor 类似地展示标题预览、更新时间、消息条数等摘要字段；具体推导算法属规划阶段任务，但产品结果须满足本架构。

## Success Criteria

- 在默认路径存在典型 Claude Code 数据时，刷新后能看到完整项目列表及各项目下的 JSONL 文件列表。
- 对 fixture 与实机样本文件，解析通过率与字段保留满足 R6：无静默丢行；未知类型可辨识；损坏行有明确报错且不拖垮整文件。
- 代码评审可通过对比 Cursor 模块的分层方式（发现 → 解析 → JSON 序列化 → 路由）确认可维护性。

## Scope Boundaries

- 第一版**不做**写入、同步到远端、与 Cursor 会话的合并检索或统一时间线。
- 第一版**不要求**支持 `~/.claude` 下除 `projects` 根目录约定以外的全部目录形态；若后续发现会话文件也在其它子路径，通过新版本扩展发现规则。
- 第一版**不要求** CLI 与 Cursor 子命令完全对称，除非实现成本极低；以 Web + API + 解析为核心。

## Key Decisions

- **默认根路径为 `~/.claude/projects`**：与用户描述一致，且与「按项目分目录」的语义对齐。
- **第一版优先级是解析语义正确**，同时 **Web 信息架构对齐 Cursor**（项目 → 会话 → 消息时间线），以降低用户心智负担并与现有 `/cursor-history` 体验一致。
- **未知 JSON 形状保留而非丢弃**：满足「完美解析」在工程上的含义——可审计、可扩展，而不是仅展示最易展示的子集；在会话/消息视图中未知项可降级为可折叠的原始片段，但不得静默丢失。

## Dependencies / Assumptions

- 假设 Claude Code 在本地写入的 JSONL 为行分隔的 JSON 文本，且单文件体量在常见开发机上可一次或分块读入；若单文件过大，在规划阶段引入流式/分页。
- 假设运行 ai2nao 服务的操作系统用户对 `~/.claude` 有读权限；权限不足时应有明确错误提示。

### Deferred to Planning

- [Needs research] 实机 `~/.claude/projects/<id>/` 下 JSONL 的**确切文件名、是否多文件、是否一文件多会话**，以及各 `type` 字段的官方或逆向稳定约定（需采样真实文件或文档）；并据此确定 **R14 会话边界的推导规则**。
- [Technical] 超大 JSONL 的内存与响应时间策略（全量解析 vs 按需行范围）。
- [Technical] API 层是否**直接复用** `cursorHistory` 的 `ChatSession` / `Message`（及摘要类型）， versus 内部 `ClaudeTranscriptEvent[]` 映射后再导出为相同 DTO；规划阶段须选定一种以兼顾类型复用与解析特异性。

## Problem Statement

ai2nao 已经能索引 Hugging Face Hub cache 里的本地模型。下一步需要把 LM Studio 下载或导入的模型也纳入本机库存：先从 LM Studio 自己的数据里读取模型下载目录，再扫描该目录下已经存在的模型，最后在 Web UI 中展示出来。

这不是新增聊天能力，也不是调用 LM Studio API 加载模型。首版目标是“看清楚本机 LM Studio 已有哪些模型，占多少空间，目录是否异常，最近同步是否成功”。

## What Makes This Cool

LM Studio 是本地模型用户最常用的入口之一。ai2nao 如果能自动读出 LM Studio 的自定义下载目录，用户不用再记路径，也不用在 Finder 里翻大型模型文件。它会把 HF cache 和 LM Studio models 放到同一个“本机模型资产视图”里，后续再做 RAG、AI 对话模型选择、磁盘治理时都有可靠数据底座。

## Existing Evidence

本机实证：

- LM Studio Electron 应用数据目录存在：`~/Library/Application Support/LM Studio/`
- 该目录下 `settings.json` 包含 `downloadsFolder`
- 当前 `downloadsFolder` 为 `/Users/quincy/.lmstudio/models`
- `~/.lmstudio/settings.json` 也存在同名字段，值同样指向 `/Users/quincy/.lmstudio/models`
- 实际模型目录形状为 `publisher/model-name/files...`
- 目录内同时存在 GGUF 单文件模型、`mmproj-*.gguf` 辅助文件、MLX/safetensors 分片模型、`config.json`、`model.safetensors.index.json`、tokenizer 和 processor 配置文件

公开资料：

- LM Studio 官方 import 文档说明，期望模型目录结构为 `~/.lmstudio/models/publisher/model/model-file.gguf`
- `lms import` 文档说明本地模型会导入到 LM Studio models directory，并支持 `--user-repo publisher/model`
- LM Studio Python 文档说明 SDK/CLI 可列出 downloaded models，但这依赖 LM Studio server/CLI 环境，不适合作为 ai2nao 首版唯一数据源

## Constraints

- 只读本机数据，不修改 LM Studio 配置、不移动模型文件、不调用删除操作。
- 优先读取 LM Studio 持久设置里的下载目录，而不是要求用户手填。
- 支持用户自定义下载目录；默认路径只是兜底。
- 支持 macOS 首版，Windows/Linux 路径可在设计中保留扩展点。
- 不把模型文件内容写入 ai2nao DB，只存路径、大小、文件类型、配置摘要、warning。
- 沿用现有 `local_inventory_sync_runs`，让同步状态与 HF 模型库存一致。
- UI 面向桌面本地浏览器，不考虑手机体验，但窄桌面窗口不能崩。
- 文件 symlink 只统计目标文件大小并记录 warning；目录 symlink 不递归跟随，避免扫描越出 models root。
- 同时发现 app settings 与 `~/.lmstudio/settings.json` 且下载目录不一致时，app settings 优先，但 API/UI 必须展示 alternative root 和 warning。

## Premises

1. LM Studio 模型目录来源应以 `settings.json.downloadsFolder` 为主，因为这是用户自定义路径的真实来源。
2. 扫描粒度应是 `publisher/model` 目录，而不是单个权重文件；同一个模型可能由多个 `.safetensors` 分片、配置文件和辅助 GGUF 组成。
3. 首版应该复用 Hugging Face 的 sync/query/route/UI 形态，而不是先抽象一个通用“所有模型来源”大平台。
4. CLI/SDK 的 `lms ls` 可作为后续增强，但首版不应依赖外部服务或 LM Studio 正在运行。

## Approaches Considered

### Approach A: 最小可维护版

Summary: 新增 `src/lmstudio` 模块，读取 LM Studio settings 中的 `downloadsFolder`，扫描 `publisher/model` 目录，落到独立 `lmstudio_models` 表，并新增 `/lmstudio-models` 页面。

Effort: M

Risk: Low

Pros:

- 复用 HF 已验证的 scan/sync/routes/queries/UI/test 模式，开发路径短。
- 不依赖 LM Studio server、CLI 或网络，用户关掉 LM Studio 也能扫描。
- 独立表避免把 HF revision 概念硬套到 LM Studio 模型目录。

Cons:

- 首版识别的是文件系统事实，不一定等同于 LM Studio UI 中“可加载”的完整语义。
- 会有少量重复 UI 代码，HF 与 LM Studio 页面暂时并列。
- Windows/Linux settings 路径如果没有样本，首版只能做保守默认。

Reuses:

- `src/huggingface/{roots,scan,sync,queries,routes}.ts`
- `local_inventory_sync_runs`
- `web/src/pages/HuggingFaceModels.tsx`
- `parseListQuery`、`formatByteSize`、现有 route lazy-loading

### Approach B: 统一本地模型库存

Summary: 建一个通用 `local_models` / `local_model_files` 数据模型，HF 和 LM Studio 都写入同一套表，UI 做成“本地模型”总览并按 source 过滤。

Effort: L

Risk: Med

Pros:

- 长期信息架构最好，用户只看到“本机模型”，不用理解来源表差异。
- 未来接 Ollama、llama.cpp、Msty 等来源时更自然。
- 可以做跨来源去重、总磁盘占用、模型格式分布。

Cons:

- 需要迁移或桥接现有 HF 表，容易把一个清晰需求扩大成库存平台重构。
- HF snapshot/revision 与 LM Studio model directory 的语义不同，过早统一会产生含混字段。
- 测试面扩大，首版 shipping 速度明显变慢。

Reuses:

- HF scanner 的 warning、size、sync-run 思路
- software inventory 的 source/filter 概念

### Approach C: 通过 `lms ls`/SDK 读取 LM Studio 模型

Summary: 不扫目录，优先调用 LM Studio CLI 或 SDK，读取 LM Studio 自己认定的 downloaded models，再把结果落库。

Effort: M

Risk: Med

Pros:

- 语义更贴近 LM Studio UI，能拿到 display name、architecture、vision、embedding 等高层信息。
- 避免自己猜测目录里的文件是否构成完整模型。
- 后续可能直接衔接“加载模型”或“打开 LM Studio”动作。

Cons:

- 依赖 `lms` 是否安装、版本是否匹配、服务是否可用，首版稳定性不如纯文件扫描。
- CLI 输出/API 可能变化，需要更多兼容层。
- 不一定能解释磁盘文件异常，例如半下载目录、孤立 mmproj、损坏分片。

Reuses:

- `~/.lmstudio/bin/lms`
- LM Studio downloaded model listing concepts

## Recommended Approach

推荐 Approach A。

原因很简单：这个项目现阶段最需要的是可靠的本机索引能力，而不是和 LM Studio 运行态深度绑定。文件系统扫描和 settings 解析能覆盖“LM Studio 不运行也能看模型”的核心场景，且能最大化复用刚落地的 HF 能力。

设计上保留两个后续出口：

- 在 `lmstudio_models` 中预留 `metadata_json`，后续可补 `lms ls` 的 display name / architecture。
- UI 文案把它叫 “LM Studio 模型”，不叫“所有本地模型”，避免过早承诺统一模型中心。

## Data Source Resolution

新增 `src/lmstudio/roots.ts`：

Resolution order:

1. API/CLI 显式传入 `root`
2. 环境变量 `LMSTUDIO_MODELS_DIR`
3. macOS app settings：`~/Library/Application Support/LM Studio/settings.json` 的 `downloadsFolder`
4. LM Studio home settings：`~/.lmstudio/settings.json` 的 `downloadsFolder`
5. 默认：`~/.lmstudio/models`

返回：

- `modelsRoot`
- `source`: `explicit | LMSTUDIO_MODELS_DIR | app_settings | home_settings | default`
- `settingsPath`: 读取成功时记录来源文件
- `alternativeRoots`: settings 冲突时列出未采用的 root
- warnings：settings 缺失、JSON 损坏、字段不是字符串、路径不存在

路径处理复用 HF 的 `expandHomePath` 思路，所有路径 resolve 成绝对路径。

## Scan Semantics

新增 `src/lmstudio/scan.ts`。

扫描规则：

- 根目录下第一层是 publisher。
- publisher 下第二层是 model directory。
- 只有模型目录下出现以下文件之一，才算一个模型：
  - `.gguf`
  - `.safetensors`
  - `model.safetensors.index.json`
  - `config.json` 且目录内存在权重文件
- 计算目录递归总大小、权重文件大小、文件数、最近修改时间。
- 识别 format：
  - `gguf`：存在主 `.gguf`
  - `mlx_safetensors`：存在 safetensors 分片且目录名或 publisher 暗示 MLX，或存在 MLX 常见配置
  - `safetensors`：存在 safetensors 但无法判定 MLX
  - `mixed`：同时存在 GGUF 和 safetensors
  - `unknown`：有配置但权重缺失
- 识别 auxiliary files：
  - `mmproj-*.gguf`
  - tokenizer/config/processor/generation/chat_template 文件
- 文件 symlink：对目标文件执行 stat，计入大小与文件列表，并记录 `file_symlink` warning。
- 目录 symlink：不进入目标目录，记录 `directory_symlink_skipped` warning。

Warnings:

- `models_root_unreadable`
- `publisher_dir_unreadable`
- `model_dir_empty`
- `model_dir_no_weight_files`
- `model_dir_malformed`
- `config_json_unreadable`
- `safetensors_index_unreadable`
- `file_stat_failed`

## Database Shape

新增 migration，例如 v14：

```sql
CREATE TABLE IF NOT EXISTS lmstudio_models (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  publisher TEXT NOT NULL,
  model_name TEXT NOT NULL,
  model_key TEXT NOT NULL,
  models_root TEXT NOT NULL,
  model_dir TEXT NOT NULL,
  format TEXT NOT NULL,
  weight_file_count INTEGER NOT NULL DEFAULT 0,
  auxiliary_file_count INTEGER NOT NULL DEFAULT 0,
  total_file_count INTEGER NOT NULL DEFAULT 0,
  total_size_bytes INTEGER NOT NULL DEFAULT 0,
  weight_size_bytes INTEGER NOT NULL DEFAULT 0,
  primary_file TEXT,
  config_json TEXT,
  warnings_json TEXT NOT NULL DEFAULT '[]',
  metadata_json TEXT NOT NULL DEFAULT '{}',
  last_modified_ms INTEGER,
  first_seen_at TEXT NOT NULL,
  last_seen_at TEXT NOT NULL,
  missing_since TEXT,
  inserted_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  UNIQUE(models_root, model_key)
);
```

`model_key` 为 `${publisher}/${model_name}`。

另建轻量文件明细表，只记录权重和关键辅助文件，不做完整文件浏览器：

```sql
CREATE TABLE IF NOT EXISTS lmstudio_model_files (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  model_id INTEGER NOT NULL REFERENCES lmstudio_models(id) ON DELETE CASCADE,
  rel_path TEXT NOT NULL,
  file_kind TEXT NOT NULL CHECK (file_kind IN ('weight', 'auxiliary')),
  format TEXT NOT NULL,
  size_bytes INTEGER NOT NULL DEFAULT 0,
  target_path TEXT,
  is_symlink INTEGER NOT NULL DEFAULT 0,
  last_modified_ms INTEGER,
  warnings_json TEXT NOT NULL DEFAULT '[]',
  inserted_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  UNIQUE(model_id, rel_path)
);
```

`local_inventory_sync_runs.source` 需要允许 `lmstudio`。

Indexes:

- `(models_root)`
- `(model_key)`
- `(missing_since)`
- `(total_size_bytes)`
- `(format)`
- `(last_seen_at)`

## API / CLI

CLI:

- `ai2nao lmstudio sync`
- `ai2nao lmstudio sync --root /path/to/models`
- `ai2nao lmstudio sync --json`

API:

- `GET /api/lmstudio/status?root=...`
- `GET /api/lmstudio/models?root=...&q=...&format=...&includeMissing=1&limit=50&offset=0`
- `POST /api/lmstudio/sync` body `{ "root": "/optional/path" }`

Status response mirrors HF:

- resolved root and source
- active/missing counts
- total disk size
- largest model
- latest sync run
- settings path used
- root resolution warnings
- alternative roots when settings disagree

## UI

新增 route:

- `/lmstudio-models`
- nav label: `LM Studio`

页面基于 `HuggingFaceModels` 改造：

- 标题：`LM Studio 模型`
- 说明：读取 LM Studio settings 中的下载目录，扫描本机模型文件，只记录元数据和大小。
- 状态卡：
  - 模型数
  - 总大小
  - 最大模型
  - models root 与来源
  - 最近同步状态
- 控件：
  - `立即同步`
  - root override input
  - 模型搜索
  - format filter
  - 显示已移除
- 列表项：
  - `publisher/model_name`
  - format badge
  - total size / weight size
  - weight files / auxiliary files / total files
  - primary file
  - last modified date
  - warning summary

空状态要区分：

- 找不到 settings：提示默认路径和可手动填 root。
- root 不存在：显示解析来源和不存在路径。
- root 存在但无模型：提示 LM Studio 期望 `publisher/model/file` 结构。

## Success Criteria

- 用户打开 `/lmstudio-models`，不用输入路径也能看到 LM Studio 当前下载目录。
- `ai2nao lmstudio sync` 能在 LM Studio 未运行时完成扫描。
- GGUF 与 safetensors/MLX 模型都不会被漏掉。
- 损坏或不完整目录不会让整次同步失败，除非 root 本身不可读。
- Web UI 与 HF 模型页面体验一致，不引入新的导航复杂度。

## What I Noticed

- 你一开始就抓住了正确的关键点：“LM Studio 下载的模型的位置是可以自定义的”。这比直接扫 `~/.lmstudio/models` 更靠谱。
- 你把需求拆成了三段：“读取下载位置 → 分辨下载模型 → UI 展示”。这正好对应 root resolution、scan/sync、presentation 三层。
- 这个功能适合沿用 HF 的形态，而不是重做平台。当前项目已经有一条成熟路径，继续铺一条并行轨道最快也最稳。
